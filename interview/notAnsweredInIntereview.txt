-> stale closure 
    - A stale closure happens when a function inside a React component uses an old value of state or props because it was created earlier and does not know about the updated value.
    - Occurs mostly in setTimeout, setInterval, event listeners, or useEffect.
-> Race conditions in frontend
    - A race condition in a frontend app happens when two or more asynchronous operations compete to update shared state, and the final outcome depends on which one finishes last — not which one was most recently requested.
    - Race conditions crop up in search/autocomplete (most common), tab switching, paginated tables, user profile switching, and any wizard where steps load async data.
-> Is Set interval risky for cooling an api ?
    - Yes, instead of that we can use setTimeout recursively to create a polling mechanism that allows us to control the timing of each API call and handle any potential errors more effectively.

-> Normalized state 
    - Normalized state is a way of structuring your app's data in the Redux/frontend store so that each piece of data is stored exactly once, and everything else just references it by ID
    - Trade off : The one downside: you have to join data back together in your selectors or components, rather than having it pre-assembled. For very simple apps this can feel like extra work. But for anything with shared entities (users appearing in posts, comments, notifications, profiles), normalization pays off quickly.

-> Websocker Architecure for a large scale app
   - WebSockets give you a persistent, full-duplex connection
   - A single WS server works fine for small apps. The problem comes when you scale horizontally: Client A is connected to Server 1, Client B is on Server 2 — if A sends a message that B should receive, Server 1 has no direct line to B's connection.The fix is a **message broker** (Redis Pub/Sub is the most common)
   - SSE ( Server-Sent Events) is unidirectional (server -> client); WebSockets are bidirectional (full-duplex).

-> The Critical Rendering Path (CRP) is the sequence of steps the browser follows to convert HTML, CSS, and JavaScript into pixels on the screen.
    - Critical Rendering Path is the set of browser steps required to render a page: parse HTML into DOM, parse CSS into CSSOM, combine them into a render tree, do layout, and then paint. CSS is render-blocking, and synchronous JavaScript can also block rendering. In frontend applications, especially React apps, we optimize CRP by reducing bundle size, deferring non-critical scripts, code splitting, lazy loading, and prioritizing above-the-fold content.
